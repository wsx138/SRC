## 一、审计摘要

对 `/upload` 头像上传功能进行代码级安全审计，共发现多类文件上传安全缺陷。修复方案采用 **纵深防御（Defense in Depth）** 策略，实现 4 层交叉校验 + 安全保存流水线，彻底消除文件上传攻击面。

---

## 二、修复前漏洞清单

| # | 漏洞 | 严重程度 | 说明 |
|---|------|----------|------|
| 1 | 无文件类型校验 | Critical | 可上传 `.py`、`.php`、`.exe` 等任意文件 |
| 2 | 原始文件名直用 | Critical | `../../etc/passwd` 路径穿越可覆盖任意文件 |
| 3 | `static/` 目录存储 | High | 上传文件可直接被 Web 服务器执行 |
| 4 | 无应用层大小限制 | Medium | 仅依赖 `MAX_CONTENT_LENGTH`，缺少细粒度控制 |
| 5 | 无文件内容校验 | High | 将 `.php` 改名为 `.jpg` 即可绕过扩展名检查 |
| 6 | 不检查 MIME | Medium | 仅信任文件名后缀，不验证 Content-Type |
| 7 | 同名覆盖 | Medium | 同名文件上传直接覆盖已有文件 |
| 8 | EXIF 元数据泄露 | Medium | 上传的原始图片可能包含 GPS、设备信息等 |

## 三、修复方案：4 层纵深防御

```
┌─────────────────────────────────────────────────┐
│              文件上传请求                          │
├─────────────────────────────────────────────────┤
│  层级 1: 扩展名白名单                             │
│         .png/.jpg/.jpeg/.gif/.webp/.bmp          │
│         拒绝 → "不支持的文件类型"                  │
├─────────────────────────────────────────────────┤
│  层级 2: Content-Type 头校验                      │
│         image/png, image/jpeg, image/gif...      │
│         拒绝 → "不支持的文件格式"                  │
├─────────────────────────────────────────────────┤
│  层级 3: 魔术字节 (Magic Bytes) 校验              │
│         imghdr.what() 读取二进制文件头             │
│         拒绝 → 即使改名 .jpg 也无法绕过            │
├─────────────────────────────────────────────────┤
│  层级 4: PIL 图片结构完整性校验                    │
│         Image.open().verify() 验证图片结构         │
│         防解压炸弹 (pixel flood)攻击              │
│         拒绝 → "图片已损坏或无法识别"              │
├─────────────────────────────────────────────────┤
│  安全保存流水线                                    │
│  ├─ PIL 重新编码 → 剥离 EXIF / 恶意 payload       │
│  ├─ UUID 随机重命名 → 防覆盖攻击                  │
│  ├─ 原子写入 (tempfile + os.replace)             │
│  └─ 非 static 目录存储 → 防直接执行               │
└─────────────────────────────────────────────────┘
```

### 层级 1：扩展名白名单

```python
ALLOWED_EXTENSIONS = frozenset({"png", "jpg", "jpeg", "gif", "webp", "bmp"})
```

使用 `frozenset`（不可变集合）防止运行时被篡改。仅允许 6 种图片扩展名。

### 层级 2：Content-Type 头部校验

```python
ALLOWED_MIMETYPES = frozenset({
    "image/png", "image/jpeg", "image/gif", "image/webp", "image/bmp",
})

content_type = file.content_type or ""
if content_type and content_type not in ALLOWED_MIMETYPES:
    return error("不支持的文件格式")
```

验证浏览器发送的 `Content-Type` 请求头。攻击者可能伪造此头部，因此**仅作为辅助校验**，不可单独依赖。

### 层级 3：魔术字节（Magic Bytes）校验

```python
import imghdr

def _check_file_type_magic(raw_data: bytes) -> str | None:
    detected = imghdr.what(None, h=raw_data[:32])
    if detected is not None and detected in {"png", "jpeg", "gif", "webp"}:
        return detected
    return None
```

通过文件二进制内容的前几个字节判断真实文件类型，**这是防改名绕过的关键**。`imghdr` 是 Python 标准库，检测以下魔术字节：

| 格式 | 魔术字节 (hex) |
|------|---------------|
| PNG | `89 50 4E 47` |
| JPEG | `FF D8 FF` |
| GIF | `47 49 46 38` |
| WebP | `52 49 46 46` (RIFF) |

**攻击者将 `shell.php` 改名为 `avatar.jpg` 上传，此层校验会拒绝**——因为文件内容前 32 字节仍然是 `<?php`，不会被识别为任何图片类型。

### 层级 4：PIL 图片结构完整性校验

```python
from PIL import Image, UnidentifiedImageError

def _is_valid_image(raw_data: bytes) -> bool:
    try:
        img = Image.open(io.BytesIO(raw_data))
        img.verify()                      # 验证结构完整性
    except (UnidentifiedImageError, Exception):
        return False

    img = Image.open(io.BytesIO(raw_data))
    width, height = img.size
    if width * height > 100_000_000:      # 防 pixel flood
        return False
    return True
```

`Image.verify()` 会验证图片文件的内部结构（如 JPEG 的 marker segments、PNG 的 chunk structure），确保文件不是"伪造头部 + 任意数据"的拼接体。

**解压炸弹防护**：限制解析后图片的总像素数 ≤ 1 亿（约 10000×10000 像素），防止攻击者上传 tiny file 爆炸为巨型位图导致服务器 OOM。

### 安全保存流水线

```python
def _sanitize_and_save(file_data: bytes, extension: str) -> str:
    img = Image.open(io.BytesIO(file_data))
    img = img.convert("RGB")          # PIL 重新编码
    
    # 原子写入: tempfile → os.replace
    tmp_fd, tmp_path = tempfile.mkstemp(dir=UPLOAD_DIR)
    os.close(tmp_fd)
    img.save(tmp_path, quality=85)
    os.replace(tmp_path, final_path)  # POSIX 原子 rename
```

**PIL 重编码的效果**：
- 剥离 EXIF/XMP/ICC 元数据（GPS 位置、相机型号、拍摄时间）
- 清除嵌入的恶意脚本（polyglot 文件：头部是图片、尾部是 PHP/JS）
- 统一编码参数（`.save()` 生产全新干净的图片文件）

**原子写入**：`tempfile.mkstemp()` + `os.replace()` 的组合确保文件写入要么完整成功、要么不出现，不会产生"写到一半"的残缺文件。

## 四、修复前后对比

| 安全维度 | 修复前 | 修复后（专业版） |
|----------|--------|-------------------|
| **扩展名** | 无限制 | 白名单 6 种 |
| **Content-Type** | 不检查 | HTTP 头白名单 |
| **魔术字节** | 不检查 | `imghdr.what()` |
| **图片结构** | 不检查 | PIL `.verify()` |
| **解压炸弹** | 无防护 | pixel 上限 1 亿 |
| **文件名** | 原始（可路径穿越） | `secure_filename()` + UUID |
| **存储位置** | `static/`（可执行） | `data/uploads/`（隔离） |
| **EXIF 隐私** | 原样保留 | PIL 重编码清除 |
| **恶意 payload** | 原样保留 | PIL 重编码清除 |
| **写入完整性** | `file.save()` | 原子写入 (`os.replace`) |
| **同名覆盖** | 覆盖 | UUID 唯一名 |
