## 一、漏洞概述

`/page` 路由通过 URL 参数 `name` 拼接文件路径读取 `pages/` 目录中的内容，但**未对用户输入做任何路径过滤或边界检查**，攻击者可通过 `../`（Linux）或 `..\`（Windows）路径穿越字符读取服务器上任意文件。

### 影响范围

| 攻击场景 | 影响 |
|----------|------|
| 读取 `/app.py` 源码 | 泄露数据库路径、密钥、业务逻辑 |
| 读取 `/data/users.db` 数据库 | 泄露所有用户密码（明文）和个人信息 |
| 读取 `/Windows/win.ini` 等系统文件 | 信息收集，为进一步攻击铺路 |

## 二、漏洞审计（白盒）

### 2.1 原始代码

```python
# app.py 第 428~461 行
@app.route("/page")
def page():
    name = request.args.get("name", "")
    # 直接拼接用户输入，无任何过滤
    filepath = os.path.join("pages", name)      # ← 路径穿越
    if os.path.exists(filepath):
        with open(filepath, "r", encoding="utf-8") as f:
            content = f.read()                  # ← 读取任意文件
```

### 2.2 漏洞根因

| 缺陷 | 说明 |
|------|------|
| **输入未过滤** | `name` 参数直接来自 URL，未做任何校验 |
| **无路径边界检查** | 未限制最终文件路径必须在 `pages/` 目录内 |
| **未使用 `abspath` 规范化** | 未将相对路径转为绝对路径后再做边界比较 |
| **`os.path.join` 不防穿越** | `os.path.join("pages", "..\\app.py")` → `pages\..\app.py` |

### 2.3 攻击原理

```
输入: name = ..\app.py

filepath = os.path.join("pages", "..\\app.py")
         = pages\..\app.py
         = app.py              ← 绕过了 pages/ 目录限制

∵ 未做路径边界检查
∴ 攻击者读取了项目根目录的 app.py 源码
```

## 三、利用验证

### POC 1：源码泄露

**攻击命令**（浏览器直接访问）：

```
http://127.0.0.1:5000/page?name=..\app.py
```

**Payload 解释**：`os.path.join("pages", "..\\app.py")` → `pages\..\app.py` → `app.py`，绕过后读取应用源码。

**实际效果** — 页面直接渲染了 `app.py` 的完整源码：

```

=====================================
安全设计说明:
--------------
1. 密码存储: 使用 werkzeug.security 的 PBKDF2 (HMAC-SHA256) 单向哈希...
import os

...
```

**截图**：`screenshots/lfi-01-source-code-leak.png`

### POC 2：数据库文件读取

**攻击命令**：

```
http://127.0.0.1:5000/page?name=..\data\users.db
```

**实际效果** — 触发 `UnicodeDecodeError`，堆栈回溯暴露了完整的服务端文件路径和调用链：


这本身也是一种信息泄露——攻击者确认了文件存在、知道了完整的路径结构。

## 四、修复方案

### 4.1 修复措施

| # | 措施 | 作用 |
|---|------|------|
| 1 | `secure_filename()` | 清除 `../`、`..\\` 等路径穿越字符 |
| 2 | `os.path.abspath()` | 将路径转为绝对路径 |
| 3 | `os.path.commonpath()` | 边界检查：确保最终路径在 `pages/` 内 |

### 4.2 修复后代码

```python
from werkzeug.utils import secure_filename

@app.route("/page")
def page():
    name = request.args.get("name", "")
    if not name:
        return render_template("index.html", page_content="请指定页面名称", user=None), 404

    # 1. 清除路径穿越字符（../ → _）
    safe_name = secure_filename(name)
    if not safe_name:
        return render_template("index.html", ..., page_content="无效的页面名称")

    # 2. 绝对路径规范化
    base_dir = os.path.abspath("pages")
    filepath = os.path.abspath(os.path.join(base_dir, safe_name))

    # 3. 边界检查：确保路径在 pages/ 目录内
    if os.path.commonpath([base_dir, filepath]) != base_dir:
        return render_template("index.html", ..., page_content="页面不存在"), 404

    if os.path.isfile(filepath):
        with open(filepath, "r", encoding="utf-8") as f:
            content = f.read()
        ...
```

### 4.3 三重防御原理

```
攻击输入: name = ..\app.py

Step 1: secure_filename("..\\app.py") → "app.py"
         ↑ ../ 和 ..\\ 被清除

Step 2: os.path.abspath(os.path.join("pages", "app.py"))
         → "D:\shixun\pages\app.py"

Step 3: os.path.commonpath(["D:\shixun\pages", "D:\shixun\pages\app.py"])
         → "D:\shixun\pages"  ✅ 匹配 base_dir → 允许
```

若攻击者绕过 Step 1（不可能，因为 `secure_filename` 会清除路径穿越字符），Step 3 的 `commonpath` 检查仍然能拦截任何越界路径。

## 五、修复验证

修复后重新执行 POC 1 payload：

```
http://127.0.0.1:5000/page?name=..\app.py
```

**结果**：页面显示 **"页面不存在"**，源码未被泄露。

**截图**：`screenshots/lfi-03-fixed-blocked.png`

正常功能未受影响——`/page?name=help` 仍能正常加载帮助中心页面。

## 六、修复前后对比

| 安全维度 | 修复前 | 修复后 |
|----------|--------|--------|
| **输入过滤** | 无 | `secure_filename()` 清除穿越字符 |
| **路径规范化** | 无 | `os.path.abspath()` 消除相对路径 |
| **边界检查** | 无 | `os.path.commonpath()` 限制在 pages/ |
| **路径穿越** | 可读取任意文件 | 已阻止 |
| **源码泄露** | app.py 完整泄露 | 无法读取 |
| **正常功能** | help 页面正常 | help 页面正常 |
