# 安全漏洞修复报告

**项目**: 用户管理系统 (Flask)  
**仓库**: https://github.com/wsx138/SRC  
**修复日期**: 2026-07-19  

---

## 漏洞清单与修复

### 1. 明文密码存储 (Critical)

| 项目 | 详情 |
|------|------|
| **文件** | `app.py` |
| **原代码** | `"password": "admin123"` / `"password": "alice2025"` |
| **风险** | 密码以明文形式存储在内存字典中，任何人获取代码即可拿到所有用户密码 |

**修复**: 引入 `werkzeug.security.generate_password_hash()` 对密码进行哈希处理，验证时使用 `check_password_hash()`，内存中不再保留原始明文密码。

```python
# 修复前
"password": "admin123"

# 修复后
"password": generate_password_hash("Admin@2025#Secure")
```

---

### 2. 弱口令 (High)

| 项目 | 详情 |
|------|------|
| **文件** | `app.py` |
| **原密码** | `admin123` / `alice2025` |
| **风险** | 极弱密码，可被暴力破解或字典攻击轻易攻破 |

**修复**: 替换为符合复杂度要求的高强度密码：
- admin: `Admin@2025#Secure`
- alice: `Alice@2025#Secure`

---

### 3. 调试注释泄露默认凭据 (High)

| 项目 | 详情 |
|------|------|
| **文件** | `templates.py/login.html` |
| **原代码** | `<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->` |
| **风险** | 任何人查看页面 HTML 源码即可直接获得管理员账号密码 |

**修复**: 删除该调试注释，不在前端暴露任何凭据信息。

---

### 4. 密码明文展示在页面 (Medium)

| 项目 | 详情 |
|------|------|
| **文件** | `templates.py/index.html` |
| **原代码** | `<span class="info-value">{{ user.password }}</span>` |
| **风险** | 用户登录后页面直接展示密码字段，即使是哈希值也不应在前端展示 |

**修复**: 从用户信息展示列表中移除密码字段。

---

### 5. 无登录失败次数限制 — 可被暴力破解 (High)

| 项目 | 详情 |
|------|------|
| **文件** | `app.py` |
| **原代码** | 登录接口无任何频率限制，可无限次尝试密码 |
| **风险** | 攻击者可通过自动化脚本不断尝试密码组合，最终暴力破解用户账号 |

**修复**: 基于 IP 的登录失败计数与封禁机制：

- 同一 IP 在 15 分钟内最多失败 5 次
- 超过限制后封禁该 IP 15 分钟，并提示剩余等待时间
- 失败时会显示剩余可尝试次数
- 登录成功后自动清除该 IP 的失败记录
- 使用 `threading.Lock` 保证线程安全

```python
MAX_FAILED_ATTEMPTS = 5       # 最大失败次数
BLOCK_DURATION = 15 * 60      # 封禁时长 15 分钟
```

## 关于 HTTPS / 抓包明文密码

**当前状态**: 抓包仍能看到明文密码。因为浏览器通过 HTTP POST 表单提交原始密码，哈希处理只在服务端进行，传输过程密码未经加密。

这是 HTTP 协议层面的问题，需要在部署层面解决：

```bash
# 开发环境临时方案：
# 将 app.py 最后一行的 app.run(...) 改为：
app.run(debug=True, host="0.0.0.0", port=5000, ssl_context="adhoc")

# 生产环境推荐方案：
# 使用 nginx 反向代理 + Let's Encrypt 免费证书启用 HTTPS
```

## 后续建议

- 将密码改用数据库存储，并使用 bcrypt / argon2 等更安全的哈希算法
