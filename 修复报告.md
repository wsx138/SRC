### 漏洞 #1: 明文密码存储

| 属性 | 内容 |
|------|------|
| **严重程度** | Critical |
| **CWE 编号** | [CWE-256: Plaintext Storage of a Password](https://cwe.mitre.org/data/definitions/256.html) |
| **OWASP 映射** | A04:2021 – Insecure Design |
| **影响文件** | `app.py` |

#### 漏洞描述

用户密码以明文字符串形式直接存储在内存字典 `USERS` 中。任何能够读取源代码或被授权访问系统内存的人，均可直接获取所有用户的原始密码。这违反了基本的安全设计原则——密码永远不应以可逆形式存储。

#### 原始代码

```python
# 修复前 — app.py
USERS = {
    "admin": {
        "username": "admin",
        "password": "admin123",        # <-- 明文密码
        "role": "admin",
        ...
    },
    "alice": {
        "username": "alice",
        "password": "alice2025",       # <-- 明文密码
        "role": "user",
        ...
    }
}

# 验证方式：直接字符串比较
if username in USERS and USERS[username]["password"] == password:
    session["username"] = username
```

#### 攻击场景

1. 开发者将代码上传至公开仓库（如 GitHub），攻击者直接获得 `admin123` / `alice2025`
2. 内部人员通过代码审查或运维访问获取所有用户密码
3. 内存转储（core dump）可泄露明文密码
4. 攻击者利用任意文件读取漏洞获取源码，进而拿到所有密码

#### 修复方案

采用 **单向哈希（One-way Hash）** 存储密码。引入 Flask 内置依赖 `werkzeug.security` 模块，使用 `generate_password_hash()` 对密码进行不可逆哈希处理，验证时通过 `check_password_hash()` 进行哈希比对。

`werkzeug.security` 内部默认使用 **PBKDF2 (HMAC-SHA256)** 算法，自带随机盐值（salt），迭代次数 260,000 次，能有效抵御彩虹表攻击和 GPU 暴力破解。

#### 修复后代码

```python
# 修复后 — app.py
from werkzeug.security import generate_password_hash, check_password_hash

# 初始化时对密码进行哈希处理，不保留明文
_INITIAL_USERS = [
    {"username": "admin", "password": "Admin@2025#Secure", "role": "admin", ...},
    {"username": "alice", "password": "Alice@2025#Secure", "role": "user", ...},
]

USERS = {}
for u in _INITIAL_USERS:
    user_data = dict(u)
    user_data["password"] = generate_password_hash(user_data["password"])
    USERS[user_data["username"]] = user_data

# 验证方式：哈希比对，不再明文比较
user = USERS.get(username)
if user and check_password_hash(user["password"], password):
    session["username"] = username
```

#### 验证方法

```bash
# 1. 启动应用后，尝试用新密码登录
curl -X POST http://localhost:5000/login \
  -d "username=admin&password=Admin@2025#Secure"

# 2. 检查 app.py 内存中的 USERS 字典——password 字段应为哈希值
#    (类似 pbkdf2:sha256:260000$... 格式)，不再是明文

# 3. 尝试用旧密码登录——应被拒绝
curl -X POST http://localhost:5000/login \
  -d "username=admin&password=admin123"
# 预期: "用户名或密码错误"
```

### 漏洞 #2: 弱口令

| 属性 | 内容 |
|------|------|
| **严重程度** | High |
| **CWE 编号** | [CWE-521: Weak Password Requirements](https://cwe.mitre.org/data/definitions/521.html) |
| **OWASP 映射** | A07:2021 – Identification and Authentication Failures |
| **影响文件** | `app.py` |

#### 漏洞描述

预设用户密码为 `admin123` 和 `alice2025`，均属于常见弱口令，已被收录于各大密码字典（如 rockyou.txt）。无任何复杂度要求、无最小长度限制。

#### 攻击场景

1. 攻击者使用字典攻击，以 `rockyou.txt` 等公开字典在数秒内即可命中
2. 使用 Hydra / Burp Suite Intruder 进行在线暴力破解
3. 结合社会工程学（已知用户名 `admin`），成功率极高

#### 修复方案

替换为符合业界最佳实践的强密码，满足以下复杂度要求：

- 最小长度 12 字符
- 包含大写字母（A-Z）
- 包含小写字母（a-z）
- 包含数字（0-9）
- 包含特殊字符（!@#$%^&* 等）

| 用户 | 旧密码（弱） | 新密码（强） |
|------|-------------|-------------|
| admin | `admin123` | `Admin@2025#Secure` |
| alice | `alice2025` | `Alice@2025#Secure` |

#### 验证方法

```bash
# 密码强度检测（以 admin 新密码为例）
# 字符集: 大写 + 小写 + 数字 + 特殊符号 = 4 类
# 长度: 17 字符
# 熵值估算: ~100 bits（远超 NIST 800-63B 推荐的 20+ bits）

# 确保旧密码无法登录
curl -X POST http://localhost:5000/login \
  -d "username=admin&password=admin123"
# 预期: "用户名或密码错误"
```

### 漏洞 #3: 调试注释泄露默认凭据

| 属性 | 内容 |
|------|------|
| **严重程度** | High |
| **CWE 编号** | [CWE-615: Inclusion of Sensitive Information in Source Code Comments](https://cwe.mitre.org/data/definitions/615.html) |
| **OWASP 映射** | A04:2021 – Insecure Design |
| **影响文件** | `templates.py/login.html` |

#### 漏洞描述

登录页面 HTML 源代码中包含调试注释，直接暴露了管理员账号密码：

```html
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

#### 攻击场景

1. 任何人打开登录页面 → 右键"查看网页源代码" → 直接看到 `admin / admin123`
2. 搜索引擎（如 Google）可能索引该页面，凭据通过搜索结果泄露
3. 浏览器开发者工具的"元素"面板直接显示注释内容
4. 自动化爬虫可批量提取 HTML 注释中的敏感信息

#### 修复方案

**彻底删除该行调试注释**，并建立开发规范：禁止在任何前端文件中以注释形式写入凭据、内部 IP、数据库连接串等敏感信息。

#### 修复前后对比

```html
<!-- 修复前 — login.html -->
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
{% extends "base.html" %}

<!-- 修复后 — login.html -->
{% extends "base.html" %}
```

#### 验证方法

```bash
# 访问登录页面，查看源码
curl http://localhost:5000/login | findstr "admin"
# 预期: 不应出现密码相关调试信息
```

### 漏洞 #4: 密码明文展示在页面

| 属性 | 内容 |
|------|------|
| **严重程度** | Medium |
| **CWE 编号** | [CWE-359: Exposure of Private Personal Information](https://cwe.mitre.org/data/definitions/359.html) |
| **OWASP 映射** | A01:2021 – Broken Access Control |
| **影响文件** | `templates.py/index.html` |

#### 漏洞描述

用户登录后的个人信息页面直接渲染了密码字段。修复前显示明文 `admin123`，修复后密码已被哈希，但仍不应在任何前端页面展示密码字段，即使它现在是哈希值。

#### 原始代码

```html
<!-- 修复前 — index.html -->
<div class="info-row">
    <span class="info-label">密码</span>
    <span class="info-value">{{ user.password }}</span>
</div>
```

#### 攻击场景

1. **肩窥攻击（Shoulder Surfing）**: 用户在公共场合登录，旁人可直接看到屏幕上的密码
2. **浏览器历史/XSS**: 页面内容被缓存或被 XSS 窃取时，密码字段一同泄露
3. **会话劫持**: 攻击者获取 Session 后访问个人信息页，直接看到哈希值（虽不可逆但增加了信息泄露面）

#### 修复方案

从个人信息展示区完全移除密码字段。密码是用户认证凭据，不是个人信息，不应在任何页面上展示。

#### 修复后代码

```html
<!-- 修复后 — index.html: 密码字段已彻底移除 -->
<div class="user-info">
    <div class="info-row"><span class="info-label">用户名</span>...</div>
    <div class="info-row"><span class="info-label">邮箱</span>...</div>
    <div class="info-row"><span class="info-label">手机</span>...</div>
    <div class="info-row"><span class="info-label">角色</span>...</div>
    <div class="info-row"><span class="info-label">余额</span>...</div>
</div>
```

#### 验证方法

```bash
# 登录后访问首页，检查页面中不再显示密码信息
curl -c cookie.txt -X POST http://localhost:5000/login \
  -d "username=admin&password=Admin@2025#Secure"
curl -b cookie.txt http://localhost:5000/ | findstr /i "password"

# 预期: 无任何输出（页面上不存在密码相关信息）
```

### 漏洞 #5: 无登录失败次数限制（暴力破解）

| 属性 | 内容 |
|------|------|
| **严重程度** | High |
| **CWE 编号** | [CWE-307: Improper Restriction of Excessive Authentication Attempts](https://cwe.mitre.org/data/definitions/307.html) |
| **OWASP 映射** | A07:2021 – Identification and Authentication Failures |
| **影响文件** | `app.py` |

#### 漏洞描述

`/login` 接口对登录失败次数没有任何限制，攻击者可以通过自动化工具无限次尝试用户名/密码组合。结合弱口令漏洞，攻击可在极短时间内成功。

#### 攻击场景

1. **字典攻击**: 使用 `hydra -l admin -P rockyou.txt http-post-form` 在数秒内命中
2. **凭据喷洒（Credential Stuffing）**: 将其他泄露数据库中获取的账号密码对批量尝试
3. **分布式攻击**: 攻击者使用代理池绕过基于 IP 的简单限制（修复方案已考虑此点）

#### 修复方案

实现基于 IP 地址的登录失败计数与封禁机制，核心设计如下：

| 参数 | 值 | 说明 |
|------|-----|------|
| `MAX_FAILED_ATTEMPTS` | 5 次 | 触发封禁的失败次数阈值 |
| `BLOCK_DURATION` | 15 分钟 | 封禁持续时长 |
| 封禁粒度 | 单 IP | 对每个来源 IP 独立计数 |
| 线程安全 | `threading.Lock` | 多线程环境下保证计数原子性 |

**设计要点**:
- 登录成功后立即清除该 IP 的失败计数（防止合法用户在封禁边缘被锁）
- 定时清理过期记录，避免内存泄漏
- 封禁时返回明确的剩余等待时间（面向用户体验）
- 失败时显示剩余可尝试次数（透明提示）

#### 修复后代码

```python
# 新增: 暴力破解防护模块

import threading

MAX_FAILED_ATTEMPTS = 5
BLOCK_DURATION = 15 * 60     # 15 分钟

_failed_logins = {}
_lock = threading.Lock()

def is_blocked(ip):
    """检查 IP 是否处于封禁期"""
    with _lock:
        rec = _failed_logins.get(ip)
        if rec and rec["count"] >= MAX_FAILED_ATTEMPTS:
            if time.time() - rec["first_fail"] <= BLOCK_DURATION:
                return True
    return False

def record_failed_attempt(ip):
    """记录一次登录失败"""
    # ...

def clear_failed_attempts(ip):
    """登录成功后清除失败记录"""
    # ...

# 在 login 路由中集成
@app.route("/login", methods=["GET", "POST"])
def login():
    if request.method == "POST":
        client_ip = request.remote_addr

        # 封禁检查
        if is_blocked(client_ip):
            return render_template("login.html",
                error="登录失败次数过多，请 N 分钟后重试")

        # 验证密码...
        if user and check_password_hash(...):
            clear_failed_attempts(client_ip)   # 成功 -> 清除
            session["username"] = username
        else:
            record_failed_attempt(client_ip)    # 失败 -> +1
```

#### 验证方法

```bash

  curl -X POST http://localhost:5000/login \
    -d "username=admin&password=wrongpass"
done

# 第 6 次应被拒绝，提示封禁信息
# 预期: "登录失败次数过多，请 15 分钟后重试"
```

### 漏洞 #6: 账号密码明文传输

| 属性 | 内容 |
|------|------|
| **严重程度** | High |
| **CWE 编号** | [CWE-319: Cleartext Transmission of Sensitive Information](https://cwe.mitre.org/data/definitions/319.html) |
| **OWASP 映射** | A02:2021 – Cryptographic Failures |
| **影响文件** | 部署配置 |

#### 漏洞描述

应用通过 HTTP（非 HTTPS）传输登录表单数据。中间人（MITM）可以在网络层截获 POST 请求体，直接读取用户输入的账号密码。注意：**服务端的密码哈希存储无法解决传输层明文问题**——密码在浏览器输入后、到达服务端之前，在网络上仍然是明文。

#### 攻击场景

1. **公共 Wi-Fi 嗅探**: 攻击者在星巴克等公共网络运行 Wireshark，捕获 HTTP POST 请求
2. **ARP 欺骗**: 局域网内通过 ARP 投毒将流量重定向至攻击者机器
3. **DNS 劫持**: 将域名解析到恶意服务器，中间人转发请求并记录密码

#### 解决方案

这是 HTTP 协议层面的问题，需在部署层面解决：

**方案 A: Flask 自签证书（仅开发环境）**

```python
if __name__ == "__main__":
    app.run(debug=True, host="0.0.0.0", port=5000, ssl_context="adhoc")
```

**方案 B: Nginx 反向代理 + Let's Encrypt（生产环境推荐）**

```nginx
server {
    listen 443 ssl;
    server_name example.com;
    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    location / {
        proxy_pass http://127.0.0.1:5000;
    }
}
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;  # 强制 HTTPS 跳转
}
```

#### 验证方法

```bash
# 修复前: HTTP 抓包可见明文
curl -X POST http://localhost:5000/login \
  -d "username=admin&password=Admin@2025#Secure"
# Wireshark 中直接可见 password=Admin@2025#Secure

# 修复后: HTTPS 抓包为密文
curl -k -X POST https://localhost:5000/login \
  -d "username=admin&password=Admin@2025#Secure"
# Wireshark 中只能看到 TLS 加密数据，无法读取内容
```

## 四、安全性提升总结

### 修复前攻击面

```
攻击者
  │
  ├─ 查看 login.html 源码 → 直接获得 admin/admin123 (漏洞#3)
  ├─ 查看 index.html 页面 → 直接展示密码 (漏洞#4)
  ├─ 获取 app.py 源码 → 获得所有明文密码 (漏洞#1)
  ├─ 字典攻击 admin123 → 秒破 (漏洞#2)
  └─ Burp Suite 暴力破解 → 无限尝试 (漏洞#5)
```

### 修复后安全防护

```
攻击者
  │
  ├─ 查看 login.html 源码 → 无任何凭据信息 (✅ 漏洞#3 已修复)
  ├─ 查看 index.html 页面 → 密码字段已移除 (✅ 漏洞#4 已修复)
  ├─ 获取 app.py 源码 → 只有哈希值，不可逆 (✅ 漏洞#1 已修复)
  ├─ 字典攻击 → 密码强度足够，字典无法命中 (✅ 漏洞#2 已修复)
  └─ 暴力破解 → 5 次后 IP 封禁 15 分钟 (✅ 漏洞#5 已修复)
```

### 遵循的安全原则

1. **纵深防御（Defense in Depth）**: 存储哈希 + 强密码 + 频率限制，多层防护
2. **最小化暴露（Least Exposure）**: 不在前端、源码中暴露任何凭据
3. **安全默认（Secure by Default）**: 密码默认哈希存储，默认强密码
4. **失效安全（Fail Secure）**: 登录失败达到阈值 → 锁定而非放行

## 五、后续改进建议

| 优先级 | 建议 | 说明 |
|--------|------|------|
| P0 | 启用 HTTPS | 解决传输层明文密码问题，防止 MITM 攻击 |
| P1 | 密码迁移至数据库 | 使用 bcrypt/argon2 替代 werkzeug 默认的 PBKDF2 |
| P2 | 多因素认证 (MFA) | 为管理员账号添加 TOTP 二次验证 |
| P3 | 密码过期策略 | 强制用户定期更换密码 |
| P3 | 账户锁定策略 | 不仅按 IP 封禁，还应锁定被攻击的账号本身 |
| P4 | 登录审计日志 | 记录所有登录尝试（成功/失败）用于事后分析 |
